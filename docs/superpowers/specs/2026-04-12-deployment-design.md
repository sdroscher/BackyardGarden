# Phase 6 — Deployment Design

**Date:** 2026-04-12
**Target platform:** Fly.io (SJC region)
**App URL:** `<appname>.fly.dev` (default, no custom domain for now)

---

## Overview

Deploy BackyardGarden to Fly.io using a `fly launch`-generated Dockerfile and `fly.toml`, with automatic migrations on deploy and GitHub Actions CI/CD (test → deploy on push to `main`).

---

## 1. Infrastructure

### Dockerfile
Generated by `fly launch --no-deploy`, producing a standard Phoenix multi-stage build:
- **Builder stage** — `elixir:slim`: fetches deps, compiles, runs `mix assets.deploy` (Tailwind + esbuild minify + phx.digest)
- **Runtime stage** — `debian:slim`: copies release artifacts only; no build tools in the final image

No custom modifications to the Dockerfile are expected beyond what `fly launch` generates.

### fly.toml
Generated by `fly launch`, then customized:
- **Region:** `sjc`
- **Machine size:** 1 shared-cpu-1x, 256MB RAM
- **Auto-stop:** enabled (saves cost when idle)
- **Health check:** HTTP GET `/` — app is healthy when the root page responds
- **Release command:** runs DB migrations before traffic switches over (see below)

Plain env vars (non-sensitive, committed in `fly.toml`):
```toml
[env]
  PHX_SERVER = "true"
  PHX_HOST   = "<appname>.fly.dev"
  DEFAULT_LOCATION = "Victoria,CA"
```

### Database
Fly Postgres already attached from Phase 5.5. `DATABASE_URL` is set as a Fly secret. The app connects to it via the standard Ecto Postgres adapter.

---

## 2. Migrations on Deploy

Add a `BackyardGarden.Release` module (`lib/backyard_garden/release.ex`) with a `migrate/0` function that runs Ecto migrations via the release binary. This is the standard Phoenix release migration pattern.

In `fly.toml`:
```toml
[deploy]
  release_command = "/app/bin/backyard_garden eval BackyardGarden.Release.migrate()"
```

Fly runs this command after the new image is built but before traffic switches over. If migrations fail, the deploy is aborted and the old version keeps serving traffic.

## 3. Secrets & Environment Variables

### Set once via `fly secrets set` (never committed):

| Secret | Source |
|---|---|
| `SECRET_KEY_BASE` | `mix phx.gen.secret` |
| `DATABASE_URL` | Auto-set by Fly Postgres attach |
| `CLOAK_KEY` | `mix run -e 'IO.puts Base.encode64(:crypto.strong_rand_bytes(32))'` |
| `AUTH0_DOMAIN` | Auth0 dashboard |
| `AUTH0_CLIENT_ID` | Auth0 dashboard |
| `AUTH0_CLIENT_SECRET` | Auth0 dashboard |
| `OPENWEATHERMAP_API_KEY` | OpenWeatherMap account |

### Auth0 callback URL update
After deploying, update the Auth0 application's **Allowed Callback URLs** to include:
```
https://<appname>.fly.dev/auth/auth0/callback
```

## 4. GitHub Actions CI/CD

File: `.github/workflows/deploy.yml`

**Trigger:** push to `main`

**Steps:**
1. Checkout code
2. Set up Elixir + Erlang (matching versions in mix.exs)
3. Install deps (`mix deps.get`)
4. Run tests (`mix test`) — uses SQLite, no external services needed
5. On success: `fly deploy --remote-only`

**GitHub secret required:** `FLY_API_TOKEN` — generate via `fly tokens create deploy`, add to repo Settings → Secrets.

`--remote-only` flag: Docker build runs on Fly's remote builders, not the GitHub Actions runner. No Docker setup or layer caching needed in the workflow.

---

## 5. Smoke Test

After first deploy:
1. Visit `https://<appname>.fly.dev` — should redirect to Auth0 login
2. Log in with Auth0 — should land on dashboard
3. Check seed library, add a planting — verify DB reads/writes work
4. Check weather card — verify `OPENWEATHERMAP_API_KEY` is set correctly
5. Verify Oban workers are running: `fly ssh console -C "/app/bin/backyard_garden eval 'Oban.check_queue(:default)'"` (should return queue info, not an error)
